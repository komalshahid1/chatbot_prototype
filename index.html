Prototype / Concept Documentation

How RAG Works

RAG in one line

RAG = Search your own business data first, then ask the AI to answer using only the matched data.

For this SaaS idea, RAG is what allows every website chatbot to answer from that specific website’s content, FAQs, pricing, refund policy, timings, and business rules.

The simple product flow

📄1. Add DataManual text, FAQs, website pages
✂️2. Chunk TextSplit long content into smaller pieces
🔢3. EmbeddingsConvert chunks into meaning-based vectors
🔍4. RetrieveFind the best chunks for the visitor question
💬5. GenerateLLM writes the final answer

Why RAG is needed

A normal AI model does not automatically know each customer’s latest website content. If a visitor asks about refund policy, prices, services, or business hours, a plain LLM may guess.

Without RAG

Visitor: “Can I get a refund?”
AI: “You may be eligible for a refund within 30 days.”

Risk: invented policy

With RAG

System first finds: “Refunds are available within 7 days only if the service has not been used.”
AI answers using that exact policy.

Better: grounded answer

Two phases of RAG

Phase A: Training / Indexing 
Business data
↓
Clean text
↓
Split into chunks
↓
Create embeddings
↓
Store in vector database

This is not fine-tuning. You are not creating a new model. You are creating a searchable knowledge base for each chatbot.

Phase B: Chat / Answering 
Visitor question
↓
Create question embedding
↓
Search matching chunks
↓
Build prompt
↓
LLM generates answer
↓
Show answer in widget

At chat time, your backend retrieves only the most relevant chunks and sends them to the AI.

Key RAG building blocks

1. Data source

For the MVP, use manual text, FAQs, and website page text. Later you can add PDFs, DOCX, sitemap crawling, and help-center articles.

2. Chunking

Large pages should be split into smaller chunks. Good starting point: 500–800 tokens per chunk with 50–100 token overlap. Good chunks improve answer quality.

Bad chunk:
Home About Pricing Contact Login Footer Copyright

Good chunk:
Refunds are available within 7 days only if the service has not been used.
3. Embeddings

Embeddings convert text into numbers so the system can search by meaning, not only exact words. This lets “money back” match “refund”, and “cost” match “pricing”.

Text:
Our Pro plan is 7000 PKR/month.

Embedding:
[0.021, -0.432, 0.118, ...]
4. Vector database

A vector database stores embeddings and finds similar chunks. Recommended MVP choice: PostgreSQL + pgvector.

source_chunks:
- workspace_id
- chatbot_id
- source_id
- content
- embedding
- metadata
5. Retriever

The retriever searches the vector database and returns the top 3–5 matching chunks for the visitor’s question.

6. Augmented prompt

The matched chunks are inserted into the prompt so the LLM has the right business context before answering.

You are a website support assistant.

Rules:
- Answer only using the provided business content.
- Do not invent prices, policies, guarantees, or timings.
- If the answer is not found, say you are not sure.

Business content:
Refunds are available within 7 days only if the service has not been used.

Visitor question:
Can I get my money back?
7. Generation

The LLM writes the final answer using the retrieved context.

Answer:
Yes. Refunds are available within 7 days, but only if the service has not been used.

Workspace-specific RAG

Every workspace must have isolated data. A visitor asking a question on Workspace A’s website should only search Workspace A’s chunks.

SELECT *
FROM source_chunks
WHERE workspace_id = :workspace_id
AND chatbot_id = :chatbot_id
ORDER BY embedding <-> :question_embedding
LIMIT 5;

This is how each customer gets their own “trained chatbot” without training a separate LLM.

End-to-end example

Business data 
Our Basic plan is 3000 PKR/month.
Our Pro plan is 7000 PKR/month.
Refunds are available within 7 days only if the service has not been used.
We are open Monday to Saturday from 10 AM to 7 PM.
Visitor question 
How much is Pro?
Retrieved chunk 
Our Pro plan is 7000 PKR/month.
AI answer 
The Pro plan is 7000 PKR per month.

Common mistakes to avoid

Mistake 1: Thinking RAG is model training

RAG is not fine-tuning. You are not changing the model. You are retrieving relevant business content at answer time.

Mistake 2: Sending the full website to AI

Do not send all website content. It is expensive, slow, and noisy. Retrieve top relevant chunks only.

Mistake 3: Bad chunks

Messy chunks create bad answers. Clean content matters more than fancy UI.

Mistake 4: No fallback

The bot must say “I’m not sure” when the answer is missing. This prevents fake pricing, fake policies, and fake guarantees.

Your first practice task

Before building the full SaaS, build one tiny RAG script:

1. Create one hardcoded business paragraph
2. Split it into 3 chunks
3. Ask a question
4. Search matching chunk
5. Send matched chunk + question to AI
6. Display answer

Then upgrade keyword search → embeddings, array storage → PostgreSQL pgvector, hardcoded text → dashboard input, playground → website widget.
